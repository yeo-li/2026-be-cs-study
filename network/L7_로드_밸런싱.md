# L7 로드 밸런싱이란?

### 1. L7 로드 밸런싱이란?

L7 로드 밸런싱은 **OSI 7계층인 애플리케이션 계층의 정보를 기준으로 요청을 여러 서버에 분산하는 방식**이다.

웹 서비스에서 L7 로드밸런서는 보통 HTTP 또는 HTTPS 요청을 이해하고, 요청 내용에 따라 적절한 백엔드 서버로 전달한다.

L7 로드밸런서가 볼 수 있는 대표적인 정보는 다음과 같다.

~~~text
Host
URL Path
HTTP Method
HTTP Header
Cookie
Query String
Content-Type
Authorization 정보 일부
gRPC service/method
~~~

이 정보들을 보고 다음처럼 요청을 나눌 수 있다.

~~~text
api.example.com      → API 서버
www.example.com      → 웹 서버
/static/*            → 정적 파일 서버
/admin/*             → 관리자 서버
Cookie: beta=true    → 베타 서버
~~~

즉, L7 로드밸런서는 단순히 IP와 포트만 보고 분산하는 것이 아니라 **요청의 의미를 읽고 분기할 수 있는 로드밸런서**다.

---

### 2. L7 로드밸런서를 사용하는 이유

L7 로드밸런서는 단순한 트래픽 분산뿐 아니라 애플리케이션 구조를 유연하게 만들기 위해 사용한다.

#### 1) 요청 내용 기반 라우팅

URL path, Host, Header, Cookie 같은 HTTP 요청 정보를 기준으로 서로 다른 서버 그룹에 요청을 보낼 수 있다.

~~~text
GET /api/users        → api-server-pool
GET /static/app.js    → static-server-pool
GET /admin/dashboard  → admin-server-pool
~~~

이 방식은 하나의 도메인과 하나의 로드밸런서 뒤에 여러 서비스를 배치할 때 유용하다.

#### 2) 마이크로서비스 분리

서비스를 기능별로 나누고, 요청 경로에 따라 각 서비스로 전달할 수 있다.

~~~text
/users    → user-service
/orders   → order-service
/payments → payment-service
~~~

클라이언트는 하나의 진입점으로 요청하지만, 내부에서는 여러 서비스가 독립적으로 요청을 처리한다.

#### 3) TLS 종료

HTTPS 요청의 암호화와 복호화를 로드밸런서에서 처리할 수 있다.

~~~text
클라이언트
   ⇄ HTTPS
L7 로드밸런서
   ⇄ HTTP 또는 HTTPS
백엔드 서버
~~~

이렇게 하면 백엔드 서버는 인증서 관리나 TLS 처리 부담을 줄이고 애플리케이션 로직에 집중할 수 있다.

#### 4) 요청/응답 수정

L7 로드밸런서는 HTTP를 이해하므로 요청이나 응답 헤더를 추가하거나 수정할 수 있다.

~~~text
X-Forwarded-For: 1.2.3.4
X-Forwarded-Proto: https
X-Forwarded-Host: example.com
~~~

백엔드 서버는 이런 헤더를 보고 원래 클라이언트 IP, 원래 프로토콜, 원래 Host를 파악할 수 있다.

#### 5) 세션 유지

쿠키나 헤더를 기준으로 같은 사용자의 요청을 같은 서버로 보내는 sticky session을 구성할 수 있다.

~~~text
Cookie: SESSION_ID=abc
   ↓
항상 web-2로 전달
~~~

서버가 세션 상태를 로컬 메모리에 저장하는 구조라면 세션 유지가 필요할 수 있다.

---

### 3. L7 로드밸런서는 프록시처럼 동작한다

L7 로드밸런서는 일반적으로 클라이언트와 서버 사이에서 HTTP 프록시처럼 동작한다.

~~~text
클라이언트
   ⇄ HTTP/HTTPS 연결 A ⇄
L7 로드밸런서
   ⇄ HTTP/HTTPS 연결 B ⇄
백엔드 서버
~~~

즉, 클라이언트는 로드밸런서와 통신하고, 로드밸런서는 백엔드 서버와 별도의 연결을 만든다.

이 구조 덕분에 L7 로드밸런서는 다음 작업을 할 수 있다.

~~~text
HTTP 요청 파싱
URL/Host/Header/Cookie 기반 라우팅
TLS 종료
HTTP Header 추가 또는 수정
Redirect 응답 생성
고정 HTTP 응답 반환
인증 연동
WAF 연동
~~~

반대로 이 말은 L7 로드밸런서가 L4 로드밸런서보다 더 많은 애플리케이션 정보를 처리한다는 뜻이다. 그만큼 기능은 풍부하지만, 단순 TCP/UDP 패킷 분산보다 처리 비용과 설정 복잡도가 커질 수 있다.

### 4. 전체 흐름 예시

다음과 같은 설정이 있다고 하자.

~~~text
Listener: HTTPS 443

Rule 1:
Host = api.example.com
Path = /v1/*
→ api-v1-target-group

Rule 2:
Host = api.example.com
Path = /v2/*
→ api-v2-target-group

Rule 3:
Host = www.example.com
→ web-target-group

Default Rule:
→ default-target-group
~~~

클라이언트 요청이 다음과 같이 들어온다.

~~~http
GET /v2/users HTTP/1.1
Host: api.example.com
Cookie: SESSION_ID=abc
~~~

L7 로드밸런서는 다음 순서로 판단한다.

~~~text
1. HTTPS 443 listener로 요청이 들어왔다.
2. TLS를 종료하고 HTTP 요청 내용을 확인한다.
3. Host가 api.example.com이다.
4. Path가 /v2/users이므로 /v2/* 규칙과 일치한다.
5. api-v2-target-group을 선택한다.
6. health check상 정상인 서버 중 하나를 선택한다.
7. 선택한 서버로 HTTP 요청을 전달한다.
~~~

전달 흐름은 다음과 같다.

~~~text
클라이언트
   ↓
https://api.example.com/v2/users
   ↓
L7 로드밸런서
   ↓
api-v2-target-group
   ↓
api-v2-2:8080
~~~

백엔드 서버 입장에서는 로드밸런서가 전달한 요청을 받는다.

~~~http
GET /v2/users HTTP/1.1
Host: api.example.com
X-Forwarded-For: 1.2.3.4
X-Forwarded-Proto: https
~~~

### 5. 핵심 요약

L7 로드밸런서는 HTTP/HTTPS 요청의 내용을 해석하고, Host, Path, Header, Cookie, Method, Query String 같은 애플리케이션 계층 정보를 기준으로 요청을 적절한 서버 그룹에
전달한다.

L4 로드밸런서가 TCP/UDP 연결을 중심으로 분산한다면, L7 로드밸런서는 HTTP 요청의 의미를 보고 분산한다.

~~~text
L4:
10.0.0.100:443/TCP로 들어왔으니 web-https-pool로 보낸다.

L7:
Host가 api.example.com이고 Path가 /v2/users이니 api-v2-target-group으로 보낸다.
~~~

L7 로드밸런서는 path 기반 라우팅, host 기반 라우팅, TLS 종료, HTTP 헤더 추가/수정, redirect, fixed response, 인증, WAF 연동 같은 기능을 제공할 수 있다. 대신 HTTP
요청을 해석하는 만큼 설정과 동작을 더 세심하게 이해해야 한다.

#### 최종 요약

L7 로드밸런서는 **Host, URL path, HTTP header, cookie 같은 애플리케이션 계층 정보를 보고 요청을 적절한 서버 그룹으로 분산하는 HTTP/HTTPS 중심의 로드밸런서**다.
