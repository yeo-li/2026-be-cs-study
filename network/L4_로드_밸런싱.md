# L4 로드 밸런싱이란?

### 1. L4 로드 밸런싱이란?

L4 로드 밸런싱은 **OSI 4계층인 전송 계층의 정보를 기준으로 TCP/UDP 연결을 여러 서버에 분산하는 방식**이다.

L4 로드밸런서는 보통 HTTP 요청의 내용까지 해석하지 않는다. 대신 다음과 같은 네트워크 연결 정보를 기준으로 요청을 처리한다.

~~~text
출발지 IP
출발지 Port
목적지 IP
목적지 Port
프로토콜: TCP 또는 UDP
연결 상태
백엔드 서버 상태
~~~

반대로 다음 정보는 일반적인 L4 로드밸런서가 판단 기준으로 사용하지 않는다.

~~~text
URL Path: /login, /api/users
HTTP Method: GET, POST
HTTP Header
Cookie
Request Body
Host Header
~~~

HTTP 요청의 경로, 헤더, 쿠키, 바디를 보고 분기하는 것은 보통 **L7 로드밸런서**의 역할이다.

---

### 2. L4 로드밸런서를 사용하는 이유

L4 로드밸런서는 주로 트래픽 분산, 고가용성, 확장성을 위해 사용한다.

#### 1) 트래픽 분산

하나의 서버가 모든 요청을 처리하지 않도록 여러 서버에 연결을 나누어 보낸다.

~~~text
클라이언트
   ↓
L4 로드밸런서
   ↓
web-1
web-2
web-3
~~~

#### 2) 고가용성

서버 하나에 장애가 발생해도 정상 서버로만 트래픽을 보내 서비스가 계속 동작하게 한다.

~~~text
web-1 정상
web-2 장애
web-3 정상
~~~

로드밸런서는 health check를 통해 장애가 발생한 `web-2`를 제외하고, `web-1`과 `web-3`에만 트래픽을 전달한다.

#### 3) 확장성

트래픽이 증가하면 서버를 추가하고, 로드밸런서가 새 서버까지 포함해 트래픽을 분산한다.

~~~text
기존: web-1, web-2
추가: web-3, web-4
~~~

### 3. L4 로드밸런서의 기본 구성

L4 로드밸런서는 보통 다음 흐름으로 동작한다.

~~~text
외부에 노출된 IP:Port
   ↓
리스너 또는 규칙
   ↓
서버 풀
   ↓
실제 서버
~~~

클라우드나 벤더마다 용어는 다르지만 구조는 비슷하다.

| 환경           | 외부 진입점                | 서버 묶음           | 실제 서버            |
|--------------|-----------------------|-----------------|------------------|
| AWS          | Listener              | Target Group    | Target           |
| Azure        | Frontend IP/Port Rule | Backend Pool    | Backend Instance |
| Google Cloud | Forwarding Rule       | Backend Service | Backend          |
| F5           | Virtual Server        | Pool            | Pool Member      |

예를 들어 하나의 VIP에 여러 포트 규칙을 둘 수 있다.

~~~text
VIP: 10.0.0.100

10.0.0.100:80/TCP    → web-http-pool
10.0.0.100:443/TCP   → web-https-pool
10.0.0.100:3306/TCP  → mysql-pool
10.0.0.100:6379/TCP  → redis-pool
~~~

클라이언트 요청이 다음과 같이 들어오면,

~~~text
목적지 IP: 10.0.0.100
목적지 Port: 443
Protocol: TCP
~~~

로드밸런서는 `10.0.0.100:443/TCP` 규칙에 맞는 `web-https-pool`을 선택한다.

즉, L4 로드밸런서는 단순히 포트 번호만 보는 것이 아니라 보통 다음 조합을 기준으로 서버 풀을 고른다.

~~~text
목적지 IP + 목적지 Port + Protocol
~~~

### 4. 서버 풀 안에서 실제 서버를 고르는 기준

서버 풀이 정해지면 로드밸런서는 그 안에서 실제 서버 하나를 선택한다.

~~~text
web-https-pool

- web-1: 10.0.1.11:443
- web-2: 10.0.1.12:443
- web-3: 10.0.1.13:443
~~~

이때 사용할 수 있는 대표적인 기준은 다음과 같다.

| 기준                   | 설명                                    |
|----------------------|---------------------------------------|
| Round Robin          | 서버를 순서대로 돌아가며 선택                      |
| Least Connections    | 현재 연결 수가 가장 적은 서버 선택                  |
| Source IP Hash       | 클라이언트 IP를 해시해 같은 클라이언트를 같은 서버로 보내기 쉬움 |
| Weighted Round Robin | 서버별 가중치를 두고 더 높은 가중치의 서버에 더 많은 트래픽 전달 |
| Health Check         | 정상으로 판단된 서버만 선택                       |

예를 들면 다음과 같은 순서로 결정된다.

~~~text
클라이언트 요청
   ↓
10.0.0.100:443
   ↓
web-https-pool 선택
   ↓
web-2:443 선택
~~~

### 5. 5-tuple과 연결 유지

L4 로드밸런서는 연결을 구분할 때 보통 다음 5가지 정보를 사용할 수 있다. 이를 **5-tuple**이라고 한다.

~~~text
Source IP
Source Port
Destination IP
Destination Port
Protocol
~~~

예를 들어 다음 연결이 있다고 하자.

~~~text
1.2.3.4:53021 → 10.0.0.100:443 TCP
~~~

이 연결이 한 번 `web-2`로 정해졌다면, 같은 TCP 연결의 후속 패킷은 계속 `web-2`로 가야 한다. 그래야 하나의 TCP 연결이 중간에 다른 서버로 흩어지지 않고 정상적으로 유지된다.

### 6. IP와 포트 변경

L4 로드밸런서는 구성에 따라 IP뿐 아니라 포트도 바꿀 수 있다.

현실에서는 앞뒤 포트를 같게 두는 경우가 많다.

~~~text
VIP:443 → backend:443
~~~

이 경우에는 포트가 그대로라서 목적지 IP만 바뀐 것처럼 보인다.

~~~text
변경 전:
1.2.3.4:53021 → 10.0.0.100:443

변경 후:
1.2.3.4:53021 → 10.0.1.12:443
~~~

하지만 다음처럼 백엔드 포트를 다르게 둘 수도 있다.

~~~text
VIP: 10.0.0.100:443

Backend:
- web-1: 10.0.1.11:8443
- web-2: 10.0.1.12:8443
~~~

이 경우 목적지 IP와 목적지 포트가 모두 바뀐다.

~~~text
변경 전:
1.2.3.4:53021 → 10.0.0.100:443

변경 후:
1.2.3.4:53021 → 10.0.1.12:8443
~~~

## 7. 응답 경로

클라이언트는 VIP와 통신한다고 생각한다.

~~~text
클라이언트 → 10.0.0.100:443
~~~

하지만 실제 서버는 `10.0.1.12:8443`일 수 있다. 이때 서버 응답은 처음에 다음 형태일 수 있다.

~~~text
출발지: 10.0.1.12:8443
목적지: 1.2.3.4:53021
~~~

로드밸런서는 응답을 다시 클라이언트가 기대하는 형태로 바꿔 보낸다.

~~~text
출발지: 10.0.0.100:443
목적지: 1.2.3.4:53021
~~~

그래야 클라이언트 입장에서는 다음과 같이 자연스럽게 보인다.

~~~text
내가 10.0.0.100:443에 요청했고,
10.0.0.100:443에서 응답을 받았다.
~~~

### 8. 와이파이 공유기와의 비교

와이파이 공유기와 L4 로드밸런서는 모두 중간에서 트래픽을 받아 다른 곳으로 보낸다는 점에서는 비슷하다.

~~~text
와이파이 공유기:
스마트폰, 노트북 → 공유기 → 인터넷

L4 로드밸런서:
외부 사용자 → L4 로드밸런서 → web-1, web-2, web-3
~~~

하지만 목적은 다르다.

| 구분         | 와이파이 공유기          | L4 로드밸런서              |
|------------|-------------------|-----------------------|
| 주 목적       | 내부 기기들이 인터넷을 쓰게 함 | 외부 요청을 여러 서버로 분산      |
| 주 방향       | 내부 → 외부           | 외부 → 내부               |
| 대상         | 휴대폰, 노트북 등        | 웹 서버, DB 서버 등         |
| 핵심 기능      | NAT, 라우팅, 포트포워딩   | 로드밸런싱, 헬스체크, 장애 서버 제외 |
| 서버 선택 알고리즘 | 보통 없음             | 있음                    |

비유하면 공유기는 내부 기기들을 인터넷으로 내보내는 중간 관리자이고, L4 로드밸런서는 외부 요청을 내부 서버들로 나눠 보내는 중간 관리자다.

### 9. 전체 흐름 예시

다음과 같은 설정이 있다고 하자.

~~~text
VIP: 10.0.0.100

10.0.0.100:443/TCP → web-https-pool

web-https-pool:
- web-1: 10.0.1.11:443
- web-2: 10.0.1.12:443
- web-3: 10.0.1.13:443
~~~

클라이언트 요청은 다음과 같이 들어온다.

~~~text
클라이언트: 1.2.3.4:53021
목적지: 10.0.0.100:443
프로토콜: TCP
~~~

로드밸런서는 다음 순서로 판단한다.

~~~text
1. 10.0.0.100:443/TCP 요청이다.
2. web-https-pool로 보내야 한다.
3. health check상 web-1, web-2, web-3 모두 정상이다.
4. 알고리즘에 따라 web-2를 선택한다.
~~~

NAT 방식이라면 전달 시 패킷 주소는 다음처럼 바뀔 수 있다.

~~~text
변경 전:
1.2.3.4:53021 → 10.0.0.100:443

변경 후:
1.2.3.4:53021 → 10.0.1.12:443
~~~

응답은 다시 클라이언트가 VIP에서 응답받은 것처럼 보이도록 바뀐다.

~~~text
변경 전:
10.0.1.12:443 → 1.2.3.4:53021

변경 후:
10.0.0.100:443 → 1.2.3.4:53021
~~~

결과적으로 클라이언트 입장에서는 계속 VIP와 통신한 것처럼 보인다.

---

### 10. 가장 짧은 한 문장 정리

L4 로드밸런서는 **목적지 IP, 목적지 포트, TCP/UDP 같은 4계층 정보를 보고 서버 풀을 고른 뒤, 정상 서버 중 하나로 연결을 분산하는 장치**이며, 구현 방식에 따라 NAT처럼 IP/Port를 바꾸거나
TCP 프록시처럼 새 연결을 만들 수 있다.
